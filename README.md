## 1. Databricks Architecture & Compute
* **Cluster Architecture:** A cluster is a group of nodes/computers working together as a single compute entity. It consists of one **Master (Driver) node** and multiple **Worker nodes**.
* **Databricks Runtime (DBR):** A virtual machine image pre-installed with standard libraries, including specific versions of Apache Spark, Scala, Python, and system drivers.
* **Photon Acceleration:** A vectorized C++ engine built by Databricks to execute SQL and DataFrame queries significantly faster than standard JVM-based Spark.
* **Autoscaling:** Dynamically scales worker nodes up or down within a user-defined minimum and maximum range based on workload demand.
* **Databricks Units (DBU):** A normalized unit of processing capability per hour used for measuring compute consumption and billing.
* **Logging:**
  * **Event Log:** Records cluster-level lifecycle events (creation, editing, autoscale triggers, termination).
  * **Driver Logs:** Captures standard output, error streams, and logging output generated by notebook execution and installed libraries.
* **Notebook Magic Commands:**
  * `%python`, `%sql`, `%scala`, `%r`: Switches cell execution language regardless of notebook default settings.
  * `%md`: Renders Markdown text (supports headers, bold/italic, lists, tables, images, and HTML).
  * `%run`: Executes another notebook sequentially within the current notebook, enabling modular code architecture.
  * `%fs`: System command shortcut for Databricks File System (DBFS) commands (e.g., `%fs ls`).
* **Databricks Utilities (`dbutils`):** Programmatic API used within Python/Scala code to interact with DBFS, secrets, widgets, and notebooks.
  * `dbutils.fs.ls()`: Lists directory contents.
  * `dbutils.secrets.get()`: Retrieves secrets securely from scope.
  * `dbutils.widgets`: Creates interactive input widgets.

## 2. Delta Lake Core Concepts
* **Overview:** Open-source storage framework that brings ACID transactions and reliability guarantees to cloud data lakes, enabling the **Lakehouse** architecture.
* **Storage Components:**
  * **Data Files:** Stored as standard, immutable Parquet files in cloud storage.
  * **Transaction Log (`_delta_log/`):** An ordered sequence of JSON commit logs serving as the single source of truth for table state.
* **Transaction Log Mechanics:**
  * Every commit generates a sequential JSON file (e.g., `000000.json`, `000001.json`).
  * Stores commit metadata, operation types (insert, update, delete), filters/predicates used, and lists of added/removed Parquet files.
  * Spark reads the transaction log to reconstruct the current valid state of the table before executing queries.
* **Key Advantages:**
  * ACID transaction support on cloud object storage.
  * Scalable metadata management.
  * Full audit history and Time Travel capabilities.
  * Built on top of open formats (Parquet + JSON).
